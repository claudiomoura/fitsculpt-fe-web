# FitSculpt Operating System

## Purpose

FitSculpt operates as an AI-first virtual company.

The company is designed to run with:
- one human decision-maker: the founder
- one command layer: HQ chat
- one execution system: Linear
- one canonical written source: `GitHub /docs`
- one memory layer: persistent memory for continuity, context, and decisions

This operating system exists to keep execution fast, clear, and low-overhead while maintaining professional standards.

## Operating Principles

- The founder makes final decisions.
- AI handles orchestration, drafting, analysis, coordination, and execution support.
- Written policy lives in `GitHub /docs`.
- Active work lives in Linear.
- Day-to-day direction happens in HQ chat.
- Important context and decisions are stored in persistent memory.
- If a system is not clearly the source of truth for something, it is not the source of truth.

## System Boundaries

### GitHub `/docs`
Use for:
- operating rules
- policies
- workflows
- templates
- decision records
- handoff formats
- beta scope
- release criteria

Do not use for:
- task-by-task execution tracking
- transient chat instructions
- unstructured notes

### Linear
Use for:
- initiatives
- projects
- issues
- current status
- priorities
- ownership
- execution sequencing
- release-related tasks

Do not use for:
- long-form policy
- canonical process documentation
- raw conversation history

### HQ Chat
Use for:
- command and control
- prioritization
- delegation
- clarifications
- daily operating decisions
- resolving ambiguity quickly

Do not use for:
- final policy storage
- durable execution history by itself

### Persistent Memory
Use for:
- key decisions
- session continuity
- non-obvious context
- discovered constraints
- recurring user preferences
- important lessons from execution

Do not use for:
- replacing formal docs
- replacing Linear issue tracking

## Source of Truth Rules

- `GitHub /docs` is the source of truth for how FitSculpt operates.
- Linear is the source of truth for what is being executed now.
- HQ chat is the source of truth for the latest instruction in motion.
- Persistent memory is the source of truth for retained context across sessions.

If conflict exists:
1. Founder decision wins.
2. Then `GitHub /docs`.
3. Then current Linear state.
4. Then memory or chat context.

## Company Structure

FitSculpt currently operates with:
- Founder
- AI Chief of Staff / Orchestrator
- AI specialists or agents as needed

This structure is intentionally lean.

## Operating Cadence

The company runs on:
- ongoing HQ chat direction
- continuous Linear execution
- session handoffs when work pauses
- weekly review
- release readiness checks before beta drops

## Documentation Standard

All docs in `/docs` should be:
- short
- operational
- current
- easy to scan
- founder-usable without translation

Every operating doc should answer:
- what this is for
- who uses it
- when it is used
- where the source of truth lives
- what good looks like

## Change Rule

When the operating model changes:
1. Update the relevant doc in `GitHub /docs`.
2. Update related templates if needed.
3. Log the decision in `docs/decision-log.md`.
4. Reflect execution changes in Linear if they affect current work.
5. Save the durable context to memory.

## Current Constraints

FitSculpt is operating under a free beta context:
- 4 weeks
- mobile-first Android APK
- first users from friends and friends-of-friends
- validation focus on usefulness, clarity, and completeness
